## Context

复习站已有 `docs/intro/README.md`（个人经历 + 后台通用心法）与 13 个专题。面试即将进行，缺一份**开场即用的自我介绍叙事脚本**。候选人背景：360 三年（智能硬件后台、DDOS 攻防、CoreDNS/Erlang/内核）+ 鹅厂六年（自研 Mesh、支付商业化、研效工具、AI 与游戏后台）。硬核弹药已散落在 xmesh-k8s、business-proxy、rate-limit、redis、agent-dev 等专题，本脚本负责把这些弹药**编排成一条"风险把控 + 稳定性"的叙事线**，并让候选人能主动控场。

约束：内容为 Markdown，随 VuePress 构建；话术须中文口语、第一人称、可直接朗读；不得堆砌自夸形容词——"聪明"必须由取舍论证和踩坑复盘自然透出。

## Goals / Non-Goals

**Goals:**
- 一篇可在面试当天照着讲的脚本，主线锁定风险把控与系统稳定性。
- 通过"埋钩子 → 引导追问 → 落到我最擅长的专题"实现候选人控场。
- 提供 60 秒电梯版 + 3~5 分钟完整版 + 引导路线图表格。
- 与既有专题双向链接，追问时可一键跳深。

**Non-Goals:**
- 不重写既有专题内容，不改动 README 既有段落语义（仅加一处入口链接）。
- 不涉及构建/CI/依赖改动。
- 不虚构履历；所有故事点取自既有专题里已沉淀的真实经历。

## Decisions

**D1 — 五段式叙事骨架（钩子→骨架→信条→埋钩子→反问），而非流水账履历。**
理由：流水账把主动权交给面试官逐条问；五段式让每段都自带"下一步引导"。副线"聪明"靠信条段的取舍论证（为什么不选 Istio、CAP 取舍）体现，比形容词可信。备选：STAR 结构——更适合单个行为问题的回答，不适合开场控场，弃用。

**D2 — 主线用"风险"串场，稳定性四板斧作为可展开的锚点。**
每个经历点都以"这里最大的风险是什么 / 我如何把控"收束（支付幂等四道闸、锁看门狗、点直购超时 PLAN A/B/C、扩缩容一致性哈希）。理由：契合用户诉求"突出风险把控、稳定性"，且这些正是候选人踩过真坑、经得起追问的点。

**D3 — 双时长版本 + 引导路线图表格。**
电梯版应对"简单介绍下自己"，完整版应对"详细说说"，路线图表格把"面试官问 X→我答 Y→抛钩子 Z"显式化，便于临场查表控场。理由：面试节奏不可控，需要可裁剪的模块。

**D4 — 交叉链接到既有专题而非复制内容。**
追问点用相对链接指向 xmesh-k8s / business-proxy / rate-limit / redis。理由：避免内容重复与漂移，保持单一事实源。

## Risks / Trade-offs

- [脚本过长、临场念不完] → 强制双版本，电梯版独立成段、路线图可查表。
- [显得刻意/背稿感] → 话术写成口语短句、留"停顿与反问"提示，强调按需裁剪而非逐字背。
- [引导钩子被面试官无视] → 路线图给多条并行路径，任一被岔开都能收回主线。
- [链接指向的专题名/路径变更] → 用站内相对链接并在 tasks 中校验目标文件存在。
- [过度自夸反噬] → spec 明确禁止形容词堆砌，聪明感只走"取舍+踩坑"通道。

## Migration Plan

新增文档，无迁移。步骤：写 `self-intro-narrative.md` → README 加入口链接 → `npm run docs:dev` 本地自查渲染与链接 → 随 main 分支正常发布。回滚：删除新文件并还原 README 的一行链接即可。
