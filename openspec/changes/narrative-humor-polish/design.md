## Context

全站 6 个内容域、约 60+ 专题，均已按 `memory-system` 强制的五段式（`::: tip 一句话结论` → 场景问题 → 实现方案 → 为什么这么做 → 为什么别的选择不行 → 沉淀结论 → `### 记忆口诀`）成文，正确性强但记忆友好度参差。每个专题配一份 `.cards.md`——**由 `scripts/generate-cards.mjs` 从源 `.md` 的 `### 记忆口诀` 段与 `## 自测` 的 `<details>` 块自动生成**，非手写。站点为 VuePress，`:::` 容器与缩进 `<details>` 块有 CI lint 门禁（`npm run check:md`，历史上多次因缩进破坏导致构建失败），另有 `npm run check:structure` 校验五段式完整。已有 `memory-system`（单篇结构标准化）、`content-redundancy-policy`（单一事实源）、`content-privacy-policy`、`interview-topic-catalog`（必含知识点）、`interview-review-hub`（首页渲染）等内容约束。

本设计要在**不动五段式骨架、不减知识点、不破构建**的前提下，为全部专题批量补齐/提质 `content-narrative-style` 规范定义的三类可记忆化元素。叙事钩子/记忆锚点的承载结构（开篇 `>`、`::: tip 一句话结论`、`### 记忆口诀`）**已存在**，本次工作重心是**质量**（叙事化 + 幽默类比），而非新增结构。

## Goals / Non-Goals

**Goals:**
- 建立一套可复制的"叙事化润色配方"，对任一专题都能机械执行、结果可验收。
- 逐域覆盖全部专题，正文锚点与 `.cards.md` 口诀双通道对齐。
- 每次改动都通过构建 + lint，红线（正确性/知识点/去重/隐私）零违反。

**Non-Goals:**
- 不新增专题、不改侧边栏/依赖图/站点结构、不重写四段式骨架。
- 不做技术内容的实质性增删（只增可读性层，不改结论）。
- 不引入新的构建工具或 markdown 插件。

## Decisions

### 决策 1：把风格规范做成"附加层"而非重写
在既有段落中**插入**钩子/类比/锚点，而非重写正文。理由：重写风险高（易引入技术错误、易破坏 `:::`/`<details>` 缩进），而插入式增量可逐段 diff 验收。**备选**：整篇重写——否决，成本与回归风险都过高。

### 决策 2：复用现有容器承载记忆元素
- 叙事钩子 → 写入 H1 后首个 `>` 引导段（已是站点惯例）。
- 幽默类比 → 放在核心机制讲解段内，用普通段落 + 一句"类比失效边界"。
- 记忆锚点 → 优先复用现有 `::: tip 一句话结论`，必要时新增一个轻量记忆块。
理由：不引入新组件、不改渲染、天然满足 `interview-review-hub` 四段式约束。**备选**：自定义 Vue 组件——否决，超出非目标且增构建复杂度。

### 决策 3：逐域分批，每域一个提交单元
按 `algo → common → internet → game-infra → game-biz → ai-llm` 顺序，每域完成后跑一次构建/lint 再进入下一域。理由：控制 blast radius，任一域出问题可独立回滚；契合仓库既有 `docs(<域>): …` 提交惯例。**备选**：一次性全量改——否决，review 与回滚困难。

### 决策 4：闪卡不手改，改源后再生成
`.cards.md` 由 `generate-cards.mjs` 从源 `.md` 的 `### 记忆口诀` + `## 自测` `<details>` 生成。因此正文锚点与闪卡口诀的一致性**不靠人工双写**：只编辑源 `.md`，每域收尾统一跑 `npm run generate:cards` 重生成并一并提交。理由：单一事实源（口诀源在 `.md`），杜绝正文↔闪卡漂移，符合 `content-redundancy-policy` 精神。**备选**：手改 `.cards.md`——否决，会被再生成覆盖且引入漂移。

### 决策 5：润色配方模板化
每个专题按固定检查清单执行：①钩子是场景非定义 ②核心难点有类比 ③类比标失效边界 ④`### 记忆口诀` 可背诵 ⑤知识点必含项自查 ⑥去重/隐私自查 ⑦本地 `check:structure` + `check:md` + 构建。理由：使"幽默"这类主观目标落到可勾选的客观项上。闪卡一致性由每域收尾的 `generate:cards` 统一保证，不进单专题清单。

## Risks / Trade-offs

- **幽默/类比不当引入技术失真** → 每个类比强制配"失效边界"一句；红线场景要求对照润色前后无新增错误。
- **破坏 `:::`/`<details>` 缩进导致 CI 构建失败** → 每域改动后本地跑构建 + 现有缩进 lint 门禁；插入式改动尽量不动既有缩进块。
- **润色稀释关键知识点** → 每专题勾选 `interview-topic-catalog` 必含项自查；只增不删。
- **风格过度/密度失衡**（笑点堆砌反而干扰阅读） → 每专题类比限"最难的 1 个核心机制"，钩子/锚点各一处，控制密度。
- **正文与闪卡口诀漂移** → 以闪卡口诀为源、正文对齐；不一致时同一提交内一并修正。

## Migration Plan

1. 先落地 `content-narrative-style` 规范到 `openspec/specs/`（归档时自动完成）。
2. 按域推进，每域一个 `docs(<域>): …` 提交；提交前对改动的源 `.md` 跑 `npm run generate:cards` 重生成闪卡，再本地 `check:structure` + `check:md` + 构建通过。
3. 回滚策略：以域为单位，任一域出问题按提交回退，不影响其他域。
4. 全域完成后跑一次全站构建，并抽查若干专题的正文锚点↔闪卡口诀一致性。
