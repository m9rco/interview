## 为什么这么做

**🌟 高性能：先问场景，再给方案**

- 秒杀、配置中心 18w QPS、序列化/反序列化压测；有些操作应下沉到客户端本地
- Kafka 架构与高性能：Rebalance、消息可靠性、顺序性
- 定位性能瓶颈的顺序：**观测→采样→分层排除**（网络/磁盘/CPU/锁）
- **四层 vs 七层负载**：为什么不选 Istio？卡在 Envoy 的 3 倍 socket 与 XDS 全量下发；更适合 HTTP，不支持点对点，跨集群还要 CROSS 模块，路由策略有限

**🌟 高可用：故障发生时可控、可隔离、可恢复**

- 请求太多？**限时、排队、削峰**
- 下游变慢？**超时、重试、熔断、隔离**
- 核心服务挂了？**降级、冗余、故障转移**
- 重复请求？**幂等、防重、状态机**
- 上线前预防？**压测、监控、混沌工程、演练、容量评估**

**🌟 分布式：为什么必须做取舍**

要快就得数据不一致——同步等所有机器变慢；要不停工就得允许延迟。**这就是 CAP。**

**分布式概念地图**

| 概念 | 解决什么问题 |
| --- | --- |
| **CAP** | 强一致 / 不卡顿 / 不怕断网 · 三选二 |
| **BASE** | 折中方案：允许中间不一致，只要**最终一致** |
| **Paxos** | 一群随时可能死机的节点，如何达成唯一共识（理论鼻祖） |
| **Raft** | Paxos 简化版：选 Leader、写流水账（工程可实现） |
| **ZAB** | ZooKeeper 专用同步协议（选主+日志复制，同类 Raft） |
| **Gossip** | 像村头大妈传八卦，一对一互传，最后全网都收到（**超大集群适用**） |
| **一致性哈希** | 扩缩容时只有极少数据需要搬家，大部队不受影响 |

**微服务四大件的职责**

| 组件 | 职责比喻 |
| --- | --- |
| RPC | **修路的**：让远端服务像本地调用一样 |
| API 网关 | **看门大爷**：鉴权、限流、路由，唯一入口 |
| 配置中心 | **全校大广播**：改配置无需重启，广播一喊立即生效 |
| 注册中心 | **动态通讯录**：谁开机就登记，谁死机就划掉 |

## 为什么别的选择不行

**游戏后台 vs 互联网后台的本质**

关注点根本不同：
- **单次响应绝对延迟** vs **整站吞吐/并发承载**
- **游戏选择"单线程无锁"**：人与人、人与怪状态高度交织，加锁代价太大；游戏是计算密集型，单线程无锁循环延迟最低
- **互联网选择"多线程/协程"**：请求彼此独立（无状态），大量时间在等 DB I/O，多协程异步能压榨吞吐

**为什么游戏单线程 / 互联网多线程**

| 维度 | 游戏后台 | 互联网后台 |
| --- | --- | --- |
| 状态耦合 | **强状态、强关联**（AOE、扣血、吸血、眩晕多线程加锁太重） | **无状态、弱关联**（甲乙买东西无本质关系） |
| 密集类型 | **CPU 计算密集**（20~60Hz Tick，视距/碰撞/伤害/寻路） | **I/O 密集**（等 MySQL、Redis、第三方 API） |
| 并发模型 | **单线程无锁 + 帧循环** | **多协程/多线程 + 异步** |
| 数据位置 | 几乎全在内存 | 主要在数据库/缓存 |

**Istio 为什么慢？**（不是控制面 Istiod，慢的是每个 Pod 的 Envoy Sidecar）

1. **协议栈"反复横跳"**：Context Switch + 数据拷贝
2. **沉重的 L7 解析**：认得 HTTP/1.1、HTTP/2、gRPC、Dubbo，能力都是代价
3. **XDS 全量下发**：500 服务 × 10 Pod = 每个 Envoy 内存维护 5000 实例；扩缩容一波爆炸

## 沉淀结论

**分布式常见坑**

- **分布式 ID**：雪花算法**时钟回拨坑**——ID 瞬间重复
- **分布式锁**：**锁提前过期坑**——业务没跑完 Redis 就放锁；需要看门狗续期
- **分布式事务**：TCC/Saga 写补偿代码 + 幂等；MQ 最终一致性最常用；强一致性能差

**HTTP/1.1 → HTTP/2 → HTTP/3 的线头阻塞演进**

- **HTTP/1.1**：一个 TCP 通道内请求串行 → 只能开多 TCP 连接（三次握手 + 慢启动 + 6 连接对 CPU/内存都是压榨）
- **HTTP/2**：多路复用 + 二进制分帧 + HPACK 头部压缩 + Server Push；一个 TCP 承载全部请求
- **痛点**：TCP 层线头阻塞——共用滑动窗口，一个包丢，整条流都停摆
- **HTTP/3**：底层换 QUIC/UDP，每个"流"独立控制，一个流丢包不影响其他流

**面试第一分钟：什么该讲，什么该藏**

- 讲**主线**：一句话概括每段经历，不背简历

- 讲**你解决了什么**：一致性、复用率、跨集群、多通道 CVM
