# 排序算法全景与选型

> 选择/插入/希尔/冒泡/归并/快排/堆排 · 稳定性 × 复杂度 × 选型公式 · 快排优化（三路 + 随机基准）· 并查集 · Unix I/O 模型速记

::: tip 一句话抓手
面试问排序，先反问四件事：**数据规模、是否近乎有序、是否大量重复、要不要稳定**。答案直接决定选型——**小/近乎有序用插入，通用用快排（工程里配三路+随机基准+小区间转插入），要稳定或链表用归并，要 Top-K / 优先级用堆**。语言标准库（Go `sort`、C++ `std::sort`）多是这些的混合体（introsort / pdqsort）。
:::

## 场景问题

拿到排序需求先自问（源自 practice `3.sorting/README`）：

1. **数据特征**：大量重复元素？（→ 三路快排）近乎有序？（→ 插入排序 O(n)）取值范围有限？（→ 计数/基数排序 O(n)）
2. **是否要稳定**：相等元素保持原相对顺序？（→ 归并、稳定插入）
3. **存储结构**：链表不适合快排（无随机访问），适合归并
4. **数据量**：小数组常数因子主导，O(n²) 的插入反而更快

## 实现方案

### 七大排序对照表

| 算法 | 平均 | 最坏 | 最好 | 空间 | 稳定 | 一句话记忆 |
| --- | --- | --- | --- | --- | --- | --- |
| 冒泡 Bubble | O(n²) | O(n²) | O(n) | O(1) | ✅ | 相邻交换，加 flag 可提前退出 |
| 选择 Selection | O(n²) | O(n²) | O(n²) | O(1) | ❌ | 每轮选最小放前面，交换次数少 |
| 插入 Insertion | O(n²) | O(n²) | **O(n)** | O(1) | ✅ | 近乎有序神器，小数组快 |
| 希尔 Shell | O(n^1.3) | O(n²) | O(n) | O(1) | ❌ | 带间隔的插入，缩小增量 |
| 归并 Merge | O(nlogn) | O(nlogn) | O(nlogn) | **O(n)** | ✅ | 稳定、可外排、适合链表 |
| 快排 Quick | O(nlogn) | **O(n²)** | O(nlogn) | O(logn) | ❌ | 通用最快，基准选不好退化 |
| 堆排 Heap | O(nlogn) | O(nlogn) | O(nlogn) | O(1) | ❌ | 原地、最坏也稳，Top-K 首选 |

线性排序（非比较，受限于取值范围）：**计数排序** O(n+k)、**基数排序** O(d·(n+k))、**桶排序** O(n)，稳定但需额外空间且只适合整数/定长 key。

### 快排的工程化优化（重点）

朴素快排在**已排序 / 大量重复**输入下退化到 O(n²)。三个关键优化：

- **随机基准 / 三数取中**：避免固定取首元素在有序输入上退化
- **三路切分（3-way，Dijkstra 荷兰国旗）**：把数组分成 `< pivot | == pivot | > pivot`，大量重复元素时把等值区一次性排除，退化为近 O(n)
- **小区间转插入排序**：子数组长度 < ~16 时切换插入，减少递归常数开销

```text
三路快排划分（大量重复元素时的杀手锏）
       lt          i        gt
  [ <pivot | ==pivot | 待定 | >pivot ]
  arr[i] < pivot → swap(i, lt), lt++, i++
  arr[i] > pivot → swap(i, gt), gt--
  arr[i] == pivot → i++
```

这正是 Go `sort` 的 pdqsort、C++ `std::sort` 的 introsort（快排 + 堆排兜底最坏 + 插入收尾）背后的思路。

### 并查集（Union-Find，practice `5.union-find`）

处理「连通性 / 分组」问题，两大优化让近乎 O(1)：

- **按秩（或按 size）合并**：小树挂大树下，控制树高
- **路径压缩**：`find` 时把沿途节点直接挂到根

```text
find(x): while x != parent[x]: parent[x] = parent[parent[x]] (路径压缩); x = parent[x]
union(a,b): 按秩把矮树根挂到高树根
```

复杂度 O(α(n))，α 是反阿克曼函数，实际 ≤ 4。用于：朋友圈/岛屿数量、Kruskal 最小生成树、判环。

### Unix I/O 模型速记（practice `0.os`）

排序之外，practice 还演示了阻塞/非阻塞/多路复用，面试常与网络编程一起问：

| 模型 | 特征 | 代表 |
| --- | --- | --- |
| 阻塞 I/O | 一直等到数据就绪，线程挂起 | `read()` 默认 |
| 非阻塞 I/O | 立即返回，需轮询 `EAGAIN` | `fcntl O_NONBLOCK` |
| I/O 多路复用 | 一个线程管多个 fd | `select/poll/epoll` |
| 信号驱动 | 就绪发信号 | `SIGIO` |
| 异步 I/O | 内核完成拷贝后通知 | `io_uring` / POSIX aio |

`select`（fd 上限 1024、每次线性扫）→ `poll`（去上限）→ `epoll`（红黑树 + 就绪链表，O(1) 就绪通知），是高并发服务器演进主线，详见 [LVS 与 epoll](/internet/lvs-epoll.md)。

## 为什么这么做

- **没有银弹**：比较排序下界是 O(nlogn)，所以“最快”取决于输入。工程库用混合策略在各种输入上都不塌方。
- **稳定性是业务需求**：多关键字排序（先按价格、价格相同保持按时间的原序）依赖稳定排序，这时归并/稳定插入不可替代。
- **并查集/路径压缩**：把「反复查连通性」从每次遍历图降到近常数，是很多图论题从超时到 AC 的关键。

## 为什么别的选择不行

- **只用快排**：面对已排序或全等元素退化 O(n²)，甚至递归爆栈 → 必须配随机基准 + 三路 + 堆排兜底。
- **只用 O(n²) 排序**：数据一大直接不可用；但小数组 / 近乎有序时插入排序常数小反而更优，所以工程库在小区间回退它。
- **线性排序当万金油**：计数/基数只在 key 取值范围可控、为整数/定长时才成立，范围一大空间爆炸，无法替代通用比较排序。

## 沉淀结论

面试速答清单：

- **选型公式**：小/近乎有序→插入；通用→快排（三路+随机基准+小区间插入）；要稳定/链表/外排→归并；Top-K/优先级→堆
- **稳定的**：冒泡、插入、归并、计数/基数/桶；**不稳定**：选择、希尔、快排、堆
- **快排最坏 O(n²)**，靠随机基准 + 三路 + introsort 兜底
- **并查集**：按秩合并 + 路径压缩 → 近 O(1)，连通性 / Kruskal / 判环
- **I/O 多路复用**：select→poll→epoll，epoll 红黑树 + 就绪链表 O(1)
