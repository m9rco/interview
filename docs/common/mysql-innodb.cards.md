# mysql-innodb — 闪卡

> InnoDB 两条主线：**索引 = B+ 树**（矮胖、叶子链表、聚簇存整行），所以「回表、覆盖索引、最左前缀、索引下推」都是围绕少读几个页；**事务 = MVCC**（undo 版本链 + ReadView 可见性判定），所以「快照读不加锁、RR 下一次生成 ReadView、间隙锁堵幻读」都是围绕一致性视图。

## 记忆口诀

- **索引本质**：B+树矮胖 / 叶子有序链表 / 树高≈IO次数
- **回表三件套**：聚簇存整行 / 二级存主键 / 覆盖索引免回表
- **命中规则**：最左前缀 / 范围列后失效 / ICP下推减回表
- **MVCC**：undo版本链 / ReadView可见性 / 快照读不加锁
- **隔离**：RC每次新建视图 / RR首次复用 / Next-Key Lock堵幻读

## Card 1

**Q**: 为什么 InnoDB 用 B+ 树做索引，而不用 B 树、红黑树或 Hash？请对比这几种结构。

**A**: B+ 树非叶子只存 key+指针扇出大、树高低（3 层≈2000 万行），IO 次数≈树高；叶子有序双向链表天然支持范围/排序。B 树节点存数据扇出小树更高；红黑树二叉树高 log₂N 且不利磁盘顺序读；Hash 等值快但不支持范围、排序、最左前缀。

## Card 2

**Q**: 什么是回表？覆盖索引如何避免它？

**A**: 二级索引叶子只存索引列+主键，查非索引列须拿主键再查一次聚簇索引取整行，即回表。若查询列全在二级索引内（含主键），无需回表，即覆盖索引，`EXPLAIN` 显示 `Using index`。

## Card 3

**Q**: 联合索引 `(a,b,c)` 在 `a=? AND b>? AND c=?` 下如何命中？

**A**: a、b 走索引，c 因 b 是范围列而无法用于索引定位（遵循最左前缀+范围列后失效）。但在 ICP 下 c 仍可在引擎层过滤减少回表。`ORDER BY` 同样遵循最左前缀，否则 `Using filesort`。

## Card 4

**Q**: MVCC 如何判断某版本对当前事务可见？

**A**: 沿 undo 版本链逐版本比对 `DB_TRX_ID` 与 ReadView：小于 `min_trx_id` 已提交可见；≥ `max_trx_id` 不可见沿 `DB_ROLL_PTR` 找旧版本；区间内则看是否在 `m_ids`（活跃事务）中，在则不可见、不在则可见。

## Card 5

**Q**: RC 与 RR 的快照读有何区别？RR 又如何压制幻读？

**A**: RC 每次快照读都新建 ReadView故可能不可重复读；RR 首次快照读生成一次后复用保证多次读一致。RR 压制幻读：快照读靠复用 ReadView 看不到新插入行；当前读靠 Next-Key Lock（记录锁+间隙锁）锁住区间阻止 INSERT。
