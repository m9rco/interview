# lockstep — 闪卡

> 只同步操作输入，靠确定性逻辑让各端算出完全一致的状态。

## 记忆口诀

**核心思想**：只传输入 / 各端同算 / 状态自洽 / 不传状态
**确定性**：定点数 / 同种子随机 / 有序遍历 / 逻辑单线程 / 帧号代时间
**等帧策略**：InputDelay 预留 / 超时补帧 HoldInput / 延迟回滚 Rollback
**vs 状态同步**：省带宽 / 客户端重算 / 强一致 / 天然录像回放

---

## Card 1

**Q**: 帧同步为什么必须用定点数，而不能直接用浮点数？

**A**: IEEE 754 浮点在不同平台/编译器/优化选项下同一运算结果可能有细微差异，帧同步要求相同输入→完全相同输出，任何微小误差都会随帧数逐渐分叉导致各端状态不一致。改用 Q16.16 定点数（整数运算）保证跨平台位级一致。

## Card 2

**Q**: 严格帧同步（Hard Lockstep）和乐观帧同步（Optimistic）有什么区别？各适合什么场景？

**A**: 严格：必须收齐本帧所有玩家输入才推进，一人卡顿全体卡顿，经典 RTS 用。乐观：超时用上一帧输入补帧先推进，收到延迟输入后回滚重算（Rollback），手感好但实现复杂，现代格斗/MOBA（如 GGPO）用。

## Card 3

**Q**: InputDelay（输入延迟）的作用是什么？调大调小分别有什么影响？

**A**: 给所有输入人为增加固定延迟帧（本地帧 N 实际执行帧 N+delay），让网络抖动时仍能在 deadline 前收齐全部输入。调大：抗抖动强但操作手感变差；调小：手感好但易卡顿。需按实际 RTT 动态调整。

## Card 4

**Q**: 帧同步如何实现断线重连和反作弊？

**A**: 重连：服务端存全量 Input 序列（帧录像），从头快速回放追帧，或用快照 + 局部回放减少追帧时间（期间跳过渲染）。反作弊：服务端跑同一份逻辑做校验，客户端每 N 帧上报关键状态 Checksum，不一致即判定作弊。

## Card 5

**Q**: 同样是多人同步，帧同步和状态同步该如何选型？

**A**: 单位多（RTS）/ 帧级判定（格斗、MOBA） 选帧同步，带宽优势碾压且强一致、天然录像。单位少 / 判定宽松（MMO、FPS） 选状态同步，客户端轻、断线直接同步当前状态。跨平台客户端需统一定点数实现，成本高时优先评估状态同步。
