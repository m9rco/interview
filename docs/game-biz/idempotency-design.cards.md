# idempotency-design — 闪卡

> 用幂等键锁定同一请求，靠去重/CAS/状态机让重复处理只生效一次。

## 记忆口诀

- **幂等键**：发起方生成 / 重试不变 / 同操作唯一
- **三道防线**：Redis SETNX 快门 / DB 唯一索引底线 / 状态机 CAS 流转
- **恰好一次**：至少一次投递 + 处理幂等 = 效果恰好一次
- **重复请求**：回放既有结果 / 幂等返回成功 / 别返回失败

## Card 1

**Q**: 幂等键为什么必须由发起方生成、并在重试中保持不变？如果服务端每次现生成会怎样？

**A**: 幂等的前提是能识别"同一个请求"。若服务端每次收到才生成键，重试就成了新请求，键不同，幂等失效、重复处理照样发生。所以客户端重试必须带同一个 `request_id`/`txn_id`。

## Card 2

**Q**: "先查后写"（if 没处理过 { 处理 }）在并发下为什么会资损？

**A**: 查和写不是原子的，中间有并发窗口。两个请求可能同时通过"没处理过"判断，各处理一次（各发一份奖）。必须靠唯一索引/CAS 把查改压成一步，消除窗口。

## Card 3

**Q**: 对比 Redis SETNX 去重令牌 与 DB 唯一索引：各自的定位是什么？为什么要叠加？

**A**: SETNX 是性能快门：高并发下先拦大部分重复、减轻 DB 压力，但会因超时/宕机/TTL 到期而漏放，不保证正确性。唯一索引是正确性底线：由 DB 存储层原子保证，兜住漏网并发，但每个重复都打到 DB。故叠加：快门挡量、索引兜底。

## Card 4

**Q**: 为什么"至少一次投递 + 处理幂等"约等于"恰好一次"？为什么不直接做精确一次投递？

**A**: 网络层无法区分"对方没收到"和"收到但 ack 丢了"，精确一次投递几乎做不到。工程解法：投递层做至少一次（重试到成功），处理层做幂等（重复无副作用），两者叠加达到效果恰好一次（effectively-once）。

## Card 5

**Q**: 状态机 + CAS 适用什么场景？纯"新增型"操作（如发一次性奖励）为什么不能只靠它？

**A**: CAS 适用有生命周期的实体（如订单），靠"当前是旧状态才能改"实现原子流转、天然幂等。但纯新增操作没有可流转的旧状态，无从 CAS，需靠唯一索引去重兜底。
