# rag-storage-cleanup — 闪卡

> **存量治理不能停服全量重建，铁律是"审计先行 + 增量操作 + 删前留证 + 用 Recall 度量"。四板斧：审计(scroll 找孤儿/过期) → 去重(DBSCAN 语义聚类保新删旧) → 修复(patch 元数据/重 embed) → 升级(Embedding 换代走蓝绿双写→后台重建→切流→删旧)。**

## 记忆口诀

- **四板斧**：审计 / 去重 / 修复 / 升级
- **四原则**：审计先行 / 增量非全量 / 删前留证 / Recall 度量
- **去重**：DBSCAN 语义聚类 / 保新删旧 / 分批小事务
- **升级**：蓝绿双写 → 后台重建 → 切流 → 删旧（全程不停服）

## Card 1

**Q**: 为什么存量治理不能"全量清空重建"？应该用什么思路替代？

**A**: 业务 7×24 在线，清空重灌会服务中断，且百万 chunk 重 embed 成本不可接受。替代思路：不停服、增量治理，以 source 为单位分批 scroll 审计 + 小事务操作。

## Card 2

**Q**: 向量不可读，如何定位"哪些 chunk 是脏的"？

**A**: 靠审计：scroll 全量扫元数据统计各来源 chunk 数与时间分布，识别孤儿（源文档已删）、过期（超 TTL/有新版本）、异常 chunk。缺血缘就先 patch 补元数据。

## Card 3

**Q**: 对比：存量去重用 **DBSCAN 语义聚类** 和 **文本 hash** 各有什么问题，为什么选前者？

**A**: 文本 hash 只拦字节完全相同的副本，改版文档（如"iOS14→iOS17"）hash 不同但语义 95% 重复会漏掉。DBSCAN 按向量密度聚类，无需预设 K，自动发现相似簇，同簇保新删旧，能识别语义重复。

## Card 4

**Q**: 对比：Embedding 模型升级时，"直接复制旧向量到新集合" 和 "蓝绿双写重建" 的差异？

**A**: 直接复制旧向量：新旧模型向量空间不一致，query 与 doc 分布不匹配导致相似度失真，结果无意义。蓝绿双写：双写→后台异步重新 embed→切查询流量→稳定后删旧集合，全程不停服且向量分布一致。

## Card 5

**Q**: 大规模删除如何防误删？清理效果怎么衡量？

**A**: 防误删：以 source 为单位增量操作、分批小事务（避免 compaction 阻塞）、删前记录 chunk ID + 原因可回溯、灰度抽样验收。度量：用固定 query 集合的 Recall@10 前后对比，而非"感觉"。
