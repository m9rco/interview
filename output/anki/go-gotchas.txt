`append` 到一个用 `a[:2]` 得到的子 slice，什么时候会改到原 slice、什么时候不会？如何强制隔离？	未超 cap 时共享底层数组，写回会污染源 slice；超 cap 触发重新分配后二者分离。想强制隔离：用 `slices.Clone`、`append([]int{}, src...)`，或三索引切片 `a[low:high:max]` 把 cap 卡死，逼下次 append 扩容。
命名返回值和匿名返回值，defer 里 `r++` 分别会不会影响最终返回？为什么？	命名返回值能改：defer 在 return 赋值之后、真正返回之前执行，改的就是那个返回变量。匿名返回值改不动：return 已把值复制到返回槽，defer 动的是局部变量。
一个返回 `error` 的函数里 `var p *MyErr = nil; return p`，调用方 `if err != nil` 为什么成立？怎么避免？	接口是 (类型, 值) 二元组，`p` 虽为 nil 但类型信息 `*MyErr` 还在，接口整体就不等于 nil。避免：成功路径直接 `return nil`（字面量，类型也为 nil），只有真出错才返回非 nil 指针。
内置 `map` 并发写会发生什么？`sync.Map` 与 `RWMutex + map` 各自的适用场景？	内置 map 并发读写触发 fatal（不可 recover）。`sync.Map` 适合读多写少 / key 稳定（read·dirty 双 map，读走无锁快路径，但无 `len`）；`RWMutex + map` 适合写频繁 / 需遍历统计，通常更快更可控。
`[]byte(s)` 常规强转会拷贝吗？为什么不能默认零拷贝共享？极致性能下怎么办？	会拷贝一次。string 不可变是 map key、并发共享、常量折叠的地基；零拷贝共享会让改 `[]byte` 反噬 string，破坏不可变契约。极致场景用 `unsafe.String` / `unsafe.Slice`（Go 1.20+）复用底层数组，但必须保证之后不改。
无缓冲 channel 和有缓冲 channel 的收发流程有何本质区别？满/空时 goroutine 去哪了？	无缓冲：`dataqsiz==0`，收发必须同时就位，发送者直接把数据拷进接收者、跳过 buf，是同步交接。有缓冲：buf 未满入队、非空出队；队列满（发送）或空（接收）时，当前 goroutine 被打包成 `sudog` 挂到 `hchan` 的 `sendq/recvq` 并 `gopark` 让出 M，对端操作时 `goready` 唤醒。
map 为什么不能取元素地址、为什么遍历无序？底层扩容怎么做的？	底层是 `bmap` 桶（每桶 8 个 kv + overflow 链）。扩容在装载因子 `count/2^B > 6.5` 或 overflow 过多时触发，桶数翻倍并渐进式搬迁（每次读写顺带迁移旧桶）。搬迁会改变元素地址 → 语言禁止 `&m[k]`；遍历时运行时随机选起始桶/cell，故意打乱顺序逼你别依赖它。
`sync.Mutex` 的正常模式和饥饿模式分别解决什么问题？何时切换？	正常模式：新 goroutine 先自旋抢锁、抢不到再排队，吞吐高但可能饿死队尾。当队头等待超过 1ms切饥饿模式：锁直接交给队头、新来的不自旋直接排队尾，保证 FIFO 不饿死；队列清空或等待 <1ms 再切回。兼顾吞吐与公平。
`recover()` 为什么必须写在 defer 里？能兜住其它 goroutine 的 panic 吗？	panic 展开栈时只保证执行 defer 链，recover 必须搭在这条路径上、且在 defer 函数中直接调用才有效（其它位置返回 nil）。它兜不住别的 goroutine：子 goroutine 的 panic 必须在其自身内部 recover，否则整个进程崩溃。常配合命名返回值把 panic 转成 error。
空接口 `interface{}` 和带方法的接口在内存里结构一样吗？`itab` 是什么？	不一样。空接口是 `eface{_type, data}`；带方法接口是 `iface{tab, data}`，多一层 `itab`——由「具体类型 × 接口类型」唯一确定、运行时缓存的方法表（含实现方法的地址）。`data` 指向堆上值副本。接口等于 nil 当且仅当 `_type`/`tab` 与 `data` 都为 nil。
