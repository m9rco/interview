为什么有状态服务(战斗服/房间服)不能像无状态服务那样起新 Pod、摘旧 Pod 就完成扩缩容？	无状态请求自包含、状态在 DB,漂移无损。有状态服务的内存态(战斗上下文、AOI、房间实时状态)不落 DB 或落得很粗,连接态是长连接需重路由,且半局状态不可重算。直接重建=数据丢失,必须走在线迁移。
主流方案"快照 + 增量 + 追平 + 切流"中,为什么先传快照还不够、必须再传增量？为什么切流前要冻结写？	快照期间源仍在处理写,快照到达目标端时已过时(stale),不追平会丢掉快照后的变更,故增量不可省。切流与最后追平之间若有新写落源,就会丢;冻结写保证切换瞬间源与目标严格一致,是零丢失的必要条件。冻结只在增量已很小时的一瞬,玩家近似无感。
增量回放为什么必须幂等？工程上怎么做到？	网络重传、协调器重试都可能让同一批 op 送达两次。做法:每个 op 带全局单调 seq,目标端记 lastApplied 去重,seq ≤ lastApplied 直接跳过;并尽量把操作设计成幂等语义(set 而非 incr),使重复应用无副作用。
对比"在线迁移"和"故障恢复(stateful-recovery)",它们面对的前提和追求的目标有何不同？	迁移面向计划内变更:源健在,可慢慢增量追平、留回滚点、失败退回,追求玩家无感。恢复面向故障:源已宕/失联,没有实时增量流,只能靠落盘的 checkpoint + WAL 回放/副本接管重建到最近可用点,追求 RPO/RTO。一句话:迁移有活源,恢复只有落盘历史。
一致性哈希扩容时如何搬迁 slot,才能不影响其余流量？为什么这样做？	逐 slot 滚动:锁定待迁 slot → 该 slot 内房间/玩家走快照+增量流程 → 切路由表 → 解锁,任一时刻只有一小撮 slot 短暂冻结。好处是把爆炸半径关进单个 slot,失败只影响局部,且可暂停/续传。
