房间推荐列表为什么用 ZSet 而不用 MySQL `ORDER BY score DESC LIMIT`？请对比两者在读、写、实时性上的差异。	ZSet 更新 member 分值是 O(log N) 内存跳表操作、读用 `ZREVRANGE` 免现算排序、内存态秒级增删；MySQL 排序列高频更新写放大、每次查询现算排序、深翻页恶化、索引维护贵，撑不住大厅量级。
融合分值由哪几个信号组成？为什么匹配度不能像热度那样直接写进全局 ZSet？	融合分 = 热度 × 时间衰减 × 匹配度。热度、新鲜度是房间自身属性可入 ZSet；匹配度依赖"谁在看"（玩家段位/模式），无法预先固化，故基础分入 ZSet 粗排、匹配度在应用层精排。
为什么必须做分值时间衰减？不做会怎样？	不衰减则早创建、一度很热的老房间永远霸榜，即便已冷清或将解散。时间衰减让分值随时间自然回落，把展示机会让给新鲜活跃的房间，与资讯流时间衰减同理。
offset 分页（`ZREVRANGE start stop`）在数据实时变动时有什么问题？如何解决？	数据重排时按 rank 偏移会漂移，翻页出现重复或漏项。解决：改用 score 游标，以"上一页最后一个分值"为界 `ZREVRANGEBYSCORE (lastScore -inf LIMIT 0 size`，避免偏移漂移。
为什么要按模式/段位分桶？脏 member（ZSet 有 ID 但详情已过期）怎么清理？	单个全局 ZSet 会成大 key（阻塞、迁移难、热点），分桶打散且天然贴合查询习惯、隐含匹配度。脏 member 三管齐下：详情 Hash 设 TTL 兜底、定时扫描 `ZREM`、读路径遇空详情顺手 `ZREM` 自愈。
