朴素 `hash(key) % N` 有什么致命缺陷？一致性哈希把重映射比例压到多少？	`% N` 在 N 变化时几乎全量重映射（缓存全失效、玩家全漂移）。一致性哈希把节点变化时的重映射压到 O(1/N)，只有落在受影响区间的 key 才迁移。
Ring Hash 里"虚拟节点（vnode）"是用来解决什么问题的？V 越大有什么影响？	解决节点少时环上分布不均。每个物理节点放 V 个 vnode，V 越大负载方差越小、越均衡（Ketama 常用 100~160 个/节点），代价是 vnode 数组内存和二分深度上升。带权只需按权分配 vnode 数。
Maglev 为什么适合 LB 数据面，而 Ring Hash 不适合？请对比这两个方案。	数据面每包都要查表：Maglev 是 `table[hash%M]` 的 O(1) 查表，Ring Hash 是 O(log(N·V)) 二分，百万 QPS 下累积成 CPU 与尾延迟。且 Maglev 后端抖动时最小扰动，连接少被打断；Ring 的 vnode 表在节点上万时更吃内存。
Jump Hash 的迁移为什么被称为"最优"？它的硬限制是什么？	桶数 `n→n+1` 时，每个 key 以恰好 1/(n+1) 概率跳到新桶，其余不动，正是理论下界。硬限制：桶必须是 `0..n-1` 连续编号，只能尾部增删桶、删不了中间桶、不支持权重。
一个需要"按机器配置带权、且随时任意上下线节点"的场景，该选哪种？为什么另外两种不行？	选 Ring Hash：原生支持带权（按权分 vnode）+ 删任意中间节点。Maglev 删中间后端要重建整表、加权不如 Ring 直接；Jump Hash 只认连续桶、不支持权重也删不了中间桶，直接出局。
